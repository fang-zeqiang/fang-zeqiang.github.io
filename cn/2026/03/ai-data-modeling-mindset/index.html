别做需求，做能力

方泽强 / 2026-03-26

分类: 数据工程, 职业发展 / 标签: AI, 数据建模, 平台思维, Agent, OpenClaw, 能力复利 / 字数: 1555

真正难的，从来不只是技术，而是人要不要换一种脑子。

前几天，一场面试进行到后半段。屏幕那头的人停了几秒，问了一句：

“你觉得，用 AI 辅助数据建模，最大的困难点是什么？”

我愣住了。

脑子里先闪过一串技术关键词，但它们都是"怎么做"的答案，不是"为什么本质上会难"的答案。

真正的答案，在我脑子里还没有成型。直到我想起了小程老师。

小程老师是我在阿里时的后端同事。年纪不大，但写代码极有章法，开会言简意赅，从不抢话。

他有一个习惯我一直记得：每次接到需求，都会先问一句——

“以后还会不会有类似的？”

如果答案是"会"，他后面的动作就完全不一样了：把实现抽成可复用的服务，写一份内部文档说明能力边界和接入方式，在周会上同步一遍让其他人知道这个轮子已经有了。

那时候我不理解。需求做完、验收通过、结项收工，不就够了？

半年后我被点醒了。

另一条业务线来了个类似需求，大家估时至少两周。小程老师看了一眼，说了句很轻的话：“用之前那套接一下，半天差不多。”

那天下午，部署日志一条条变绿。原本要排期、协调、开发、联调的一整套流程，被他之前沉淀下来的能力，压缩成了半天。

站在那一刻，我突然明白了一件事：

需求解决的是今天，能力购买的是明天。

这件事后来改变了我的工作方式。

最近我开始用 Cursor 和 Claude 辅助做数据建模。第一次上手，花了整整两天。真正耗时间的不是建了几张表，而是把 DWD / DWS / ADS 的分层逻辑一层层梳理清楚——备菜、炒菜、上菜，每一层该干什么、不该干什么，把最容易踩坑的地方掰开揉碎，写成一份可执行的规范。

如果只为交差，几段 SQL 也能跑，报表也未必不能看。

但我越来越清楚：先跑起来再说的做法，上线就是终点。而如果多走一步，这一次的投入就能变成下一次的起点。

于是我做了和小程老师一样的选择：把方法论整理成模板，把常见错误写成 Checklist，最后封装成一个 OpenClaw Skill。

这个 Skill 不是什么宏大的产品，更像是一个数据工程师在长期实战里磨出来的一组原子能力。比如产品经理说一句"帮我看看这周销售为什么掉了"，背后可能需要多周期聚合、维度交叉、口径校验三个能力协同工作——这些手法，就是我沉淀在里面的东西。

国内大厂的数据同学几乎天天在用这些手法。但放到全球开源社区里，真正系统、完整、可直接参考的文档，反而没有想象中那么多。

真正的复利，不是把一件事做得更快，而是让同一份能力被反复调用。

在 Agent 时代，这件事有了更大的意义。

你完全可以想象这样一条链路：产品经理说了一句"帮我看看这周销售为什么掉了"。某个 Agent 调用你的多周期聚合能力，另一个调用你的维度交叉能力，再一个调用你的口径校验能力，最后拼出一份可读、可追溯、可解释的分析结论。

在这个过程中，用户也许根本不知道这些能力是谁写的。但那并不重要——只要你的能力被调用，价值就已经发生了。

AI 让写代码、写 SQL 的边际成本迅速下降。“会做"这件事本身，正在变得不稀缺。真正开始变贵的，是你能不能从一个具体问题里，抽象出一个别人以后还能继续用的解法。

与其把自己活成被需求追着跑的消防员，不如把自己练成一座稳定供电的变电站。

所以，如果现在让我重新回答那道面试题——

最大的困难不是 AI 能不能写 SQL，而是人愿不愿意把"完成需求"的脑子，换成"创造能力"的脑子。

小程老师很早就换了。我正在换。

不要做需求的奴隶，要做能力的投资者。
